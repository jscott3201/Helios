# M07 In-Memory KV Cache Core Evidence

This file records the local M07 work completed on 2026-06-30.

## Status

M07 is implemented as a Rust in-memory RAM prefix cache core with deterministic namespace hashing, logical KV block IDs, Gemma 4 layer block metadata, native-managed logical handles, RAM LRU eviction by byte budget, copy-on-write conversation fork metadata, restore-vs-fresh tests, wrong-namespace/corruption rejection, and TUI-visible cache accounting.

SSD persistence is intentionally disabled for this milestone.

## Implemented

- Added `gemma4d-kv` block/key types: `KvNamespace`, `NamespaceHash`, `KvBlockKey`, `BlockId`, `LayerBlockMetadata`, `RamPrefixBlock`, `RestoredPrefix`, and `NativeLogicalHandle`.
- Added namespace hashing over model, revision, weights, quantization, tokenizer, chat template, prompt token prefix, raw prompt, adapter identity/hash, KV layout, cache mode, MLX version, and engine version.
- Added RAM LRU residency policy with byte budget, resident byte/block counts, hits, misses, evictions, restore failures, hit rate, and explicit `ssd_enabled=false` accounting.
- Added checksum validation for stored RAM prefix payloads.
- Added copy-on-write conversation fork metadata for shared prefix blocks and private suffix blocks.
- Added fixture restore-vs-fresh coverage for 1K, 4K, 8K, and 16K contexts.
- Added wrong model/template/prompt hash rejection checks in the M07 evidence runner.
- Added M07 cache accounting DTO support in the TUI provider and cache page.

## Caveat

The M07 restore matrix uses deterministic fixture observations rather than full real-model tensor replay. Native-managed logical handles are present, but native tensor export/import remains a later integration step after the native KV ABI stabilizes.
